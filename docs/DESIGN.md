# Design — Layer 1 Inspection Tools

> Companion to [`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md), which
> explains *why* the namespace landscape looks the way it does. This document
> explains *how* the three tools in this repository navigate it.

The three tools shipped today — `namespace_detect`, `list_products`,
`get_product_details` — are deliberately a **small, read-only set**. They
exist to answer one question well: *"What is in this Comms Cloud org, and
how is it structured?"*

This document covers:

- Common design principles applied across all three tools
- Per-tool design (purpose, SOQL strategy, returned shape, known
  edge cases)
- Tool selection guidance
- What is intentionally *not* in Layer 1

Concrete request/response examples live in [`EXAMPLES.md`](./EXAMPLES.md)
(to be added).

---

## 1. Scope and non-goals

### In scope

- **Read-only inspection** of Communications Cloud assets across:
  - Vlocity CMT (`vlocity_cmt__*`)
  - Core / Industries (standard NS, including Comms-on-Core / Revenue Cloud)
  - Hybrid orgs that have both at once
- Functional surface: catalogs, categories, products, product
  attributes, ObjectClass hierarchies
- Org connectivity via the Salesforce CLI's existing authenticated
  sessions — no separate credential handling in this server

### Explicitly NOT in scope (for Layer 1)

- Any write or delete operation
- Catalog or asset *generation* from specification documents
- Migration *execution* between namespaces
- Pricing details beyond what is needed to confirm a product is for
  sale (i.e. no `PriceListEntry` walking, no
  `PriceAdjustmentSchedule` evaluation)
- Order Management runtime objects (`OrchestrationPlan__c`,
  fulfillment progress, etc.)
- Promotions, time plans, advanced rules execution

These belong to subsequent layers and are intentionally kept out of the
open MVP. See [`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md) §5
for context.

## 2. Common design principles

### 2.1 Namespace-transparent dispatch

Every tool starts by detecting which namespaces are *available* in the
target org (not just "is one of them present", but specifically which
probe objects can be described). The same set of probes is used across
all three tools so the answer is consistent:

| Probe object | Implies |
|--------------|---------|
| `vlocity_cmt__Catalog__c` | Vlocity CMT package is installed |
| `ProductCatalog` | Core PCM is provisioned |
| `ProductCategory` | Core PCM is provisioned |
| `AttributeDefinition` | Core PCM is provisioned |

The host org is then classified as one of `vlocity_cmt`, `standard`,
`hybrid`, or `none`. Subsequent SOQL is generated by branching on this
classification — there is no global "mode flag" that the caller has to
pick. Callers can override with an explicit `mode` argument when they
want to force one side on a hybrid org.

For hybrid orgs in `mode: "auto"`, results include **both** views,
clearly labelled. This is the only honest answer: a hybrid org genuinely
has two parallel representations of the same products, and the caller
should see both.

### 2.2 Defensive querying

The standard NS landscape is still evolving, and we have observed
material variability across orgs that all claim to be "standard NS":

- `ProductCatalog` field sets differ (some orgs have `Code`,
  `Description`, `EffectiveStartDate`; others have only `Name`).
- `ProductAttributeDefinition` and `AttributeCategory` exist in some
  Core orgs but not all hybrid orgs.
- Vlocity package versions add or rename objects
  (`vlocity_cmt__ObjectClass__c` vs `vlocity_cmt__ObjectType__c`).

The tools therefore:

- Query only universally-available fields by default
  (`SELECT Id, Name` on `ProductCatalog`, for example).
- Re-check object existence inside a tool when a downstream step
  depends on a less-universal sObject (e.g. `M3` re-checks for
  `ProductAttributeDefinition` and returns a graceful "no attributes
  in this org" result if missing).
- Filter out null foreign keys before composing `IN` clauses
  (e.g. `vlocity_cmt__Product2Id__c IS NULL` rows are excluded from
  `M2`).

This costs an extra describe call here and there. In return, the same
tool binary works against legacy CMT orgs, full Core orgs, and
half-migrated hybrids without configuration.

### 2.3 SOQL safety

All record IDs that are interpolated into SOQL must pass a 15- or
18-character alphanumeric check (`isValidSalesforceId`). Any input
that fails this check is rejected before the query is built. This is
the only place where user input becomes part of SOQL; everything else
is constant text.

### 2.4 Connection management

Salesforce connections are obtained via `@salesforce/core`'s `Org`
class, keyed by alias or username that the Salesforce CLI already
knows about. There is **no credential handling in this server**:
authentication is delegated entirely to `sf org login web`.

Connections are cached per process to avoid re-resolving the same alias
on each tool call.

### 2.5 Tool interface

All tools share the same shape:

- Inputs declared via a `zod` schema (so the MCP server publishes a
  JSON schema for IDE / LLM consumption)
- Single text content block returned with pretty-printed JSON
- An `isError` flag flips to `true` for tool-level errors with a
  human-readable message; the JSON shape is never half-filled

This keeps results easy for an LLM to consume and easy for a human to
read in raw form.

## 3. Tool: `namespace_detect` (M1)

### Purpose

Answer the very first question any other tool needs to answer: *what
kind of Comms Cloud org am I looking at?* Without this, every other
SOQL strategy is a guess.

### Inputs

- `targetOrg?: string` — alias or username, defaults to the CLI's
  default target

### SOQL / describe strategy

- Six `describe` calls in parallel against probe objects (three on the
  Vlocity side, three on the Core side)
- Classification:
  - Vlocity probe AND all-Core-probes-pass → `hybrid`
  - Vlocity probe only → `vlocity_cmt`
  - All Core probes pass only → `standard`
  - Else → `none`

### Returned shape

The classification result is paired with the **raw probe outcomes**
(boolean per object). This is deliberate: the caller should be able to
see *why* the tool classified the org the way it did, so disagreements
between this tool and the user's mental model are easy to resolve
(e.g. "you said `standard`, but actually `AttributeDefinition` was
missing because of a permission set issue, not a missing package").

### Edge cases handled

- Older Vlocity versions ship `ObjectClass__c` but not
  `ObjectType__c`. `Catalog__c` is the primary positive signal.
- Orgs that *almost* look like Core (have `ProductCatalog` but not
  `AttributeDefinition`) are reported as `none` for the standard side
  to avoid downstream tools building Core-shaped queries against an
  incomplete schema.

## 4. Tool: `list_products` (M2)

### Purpose

Walk the catalog hierarchy. Surfaces "what products live in this org,
under what catalog / category". For a hybrid org, surfaces both views
side by side.

### Inputs

- `targetOrg?: string`
- `mode?: "auto" | "vlocity_cmt" | "standard"` — defaults to `auto`
- `catalogId?: string` — scope to a single catalog
- `includeInactive?: boolean` — defaults to `false` for the Core side
- `productLimit?: number` — per-catalog cap, defaults to 200

### SOQL strategy — Vlocity side

```
1. Catalogs:
   SELECT Id, Name, vlocity_cmt__CatalogCode__c
   FROM vlocity_cmt__Catalog__c
   [WHERE Id = :catalogId]

2. Catalog ↔ Product relationships (junction):
   SELECT vlocity_cmt__CatalogId__c, vlocity_cmt__Product2Id__c
   FROM vlocity_cmt__CatalogProductRelationship__c
   WHERE vlocity_cmt__CatalogId__c IN :catalogIds
     AND vlocity_cmt__Product2Id__c != null

3. Products (resolve relationships):
   SELECT Id, Name, ProductCode, IsActive,
          vlocity_cmt__ObjectTypeId__c
   FROM Product2
   WHERE Id IN :productIds

4. ObjectClass hierarchy (independent dimension):
   SELECT Id, Name, vlocity_cmt__ParentObjectClassId__c
   FROM vlocity_cmt__ObjectClass__c
   ORDER BY Name
   → build tree client-side via parent IDs
```

Per [`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md) §4.3, Vlocity
treats *Catalog ↔ Product* and *ObjectClass tree* as two **independent
dimensions**. A product can belong to a catalog without being placed in
an ObjectClass, and vice versa. The tool reflects this by returning
catalogs (each carrying its own product list) and the ObjectClass tree
as two top-level fields.

### SOQL strategy — Core / standard side

```
1. Catalogs:
   SELECT Id, Name FROM ProductCatalog [WHERE Id = :catalogId]
   (intentionally minimal field set per §2.2)

2. Categories (one query for all selected catalogs):
   SELECT Id, Name, CatalogId, ParentCategoryId
   FROM ProductCategory
   WHERE CatalogId IN :catalogIds

3. Product ↔ Category links (single junction):
   SELECT ProductId, ProductCategoryId, CatalogId
   FROM ProductCategoryProduct
   WHERE CatalogId IN :catalogIds

4. Products:
   SELECT Id, Name, ProductCode, IsActive
   FROM Product2
   WHERE Id IN :productIds
   [AND IsActive = true]
```

The Core hierarchy is **uniform**: every product reaches its catalog
through a category. The tool builds a per-catalog tree client-side
from `ParentCategoryId`, attaches products to their categories via
`ProductCategoryProduct`, and surfaces any catalog-level (no-category)
products under a synthetic "Uncategorized" node so nothing is silently
dropped.

### Returned shape (sketch)

```jsonc
{
  "detectedNamespace": "hybrid",
  "requestedMode": "auto",
  "queried": ["vlocity_cmt", "standard"],
  "details": { ... },
  "vlocity_cmt": {
    "catalogs": [ { "id", "name", "code", "products": [...] } ],
    "objectClasses": [ /* nested tree */ ]
  },
  "standard": {
    "catalogs": [ { "id", "name", "categories": [ /* nested */ ] } ]
  }
}
```

### Edge cases handled

- Catalog-product rows with `Product2Id__c IS NULL` (these represent
  promotions rather than products and would crash the downstream `IN`
  clause if not filtered).
- Cross-catalog category parent links (a `ParentCategoryId` pointing
  to a category in a different catalog is treated as a tree root in
  the current catalog rather than spanning catalogs silently).
- Catalogs with zero products (returned with an empty `products` list
  rather than being omitted, so the caller can see "this catalog
  exists but is empty").

### What this tool is *not* designed to do

- Compute price for a product (no `PriceListEntry__c` /
  `PriceBookEntry` traversal)
- Evaluate qualification rules / context rules
- Resolve bundle structure recursively (Layer 1 stops at the
  product level)

## 5. Tool: `get_product_details` (M3)

### Purpose

Once `list_products` has surfaced a candidate product, this tool
returns the full picture: ObjectClass lineage, parsed per-product
attribute BLOB (Vlocity), Core-style attribute definitions, and the
catalog memberships.

### Inputs

- `productId: string` — required, 15- or 18-character Salesforce ID
- `targetOrg?: string`
- `mode?: "auto" | "vlocity_cmt" | "standard"`
- `includeRawJsonAttribute?: boolean` — return the original
  `JSONAttribute__c` text verbatim alongside the parsed view (debug
  aid)

### SOQL strategy — Vlocity side

```
1. Product2 core + vlocity_cmt extensions:
   SELECT Id, Name, ProductCode, Description, Family, IsActive,
          vlocity_cmt__SpecificationType__c,
          vlocity_cmt__SubType__c,
          vlocity_cmt__Status__c,
          vlocity_cmt__IsOrderable__c,
          vlocity_cmt__IsConfigurable__c,
          vlocity_cmt__GlobalKey__c,
          vlocity_cmt__ObjectTypeId__c,
          vlocity_cmt__JSONAttribute__c,
          vlocity_cmt__AttributeMetadata__c
   FROM Product2 WHERE Id = :productId

2. ObjectClass parent chain (recursive, depth capped at 10):
   SELECT Id, Name, vlocity_cmt__ParentObjectClassId__c
   FROM vlocity_cmt__ObjectClass__c WHERE Id = :currentId
   → repeat until ParentObjectClassId__c is null

3. Parse vlocity_cmt__JSONAttribute__c (see §5.3)
```

### SOQL strategy — Core / standard side

```
1. Product2 core:
   SELECT Id, Name, ProductCode, Description, Family, IsActive
   FROM Product2 WHERE Id = :productId

2. Catalog / category memberships:
   SELECT ProductCategoryId, CatalogId
   FROM ProductCategoryProduct
   WHERE ProductId = :productId
   → then resolve Category.Name and Catalog.Name in two batch queries

3. Attribute definitions (if available in this org):
   First: sObjectExists("ProductAttributeDefinition")
   If yes:
     SELECT Id, AttributeDefinitionId, AttributeCategoryId
     FROM ProductAttributeDefinition WHERE Product2Id = :productId
     → batch fetch AttributeDefinition details
     → batch fetch AttributeCategory names if that object exists
   If no: return empty attributes list with the product still populated
```

### 5.3 Parsing `vlocity_cmt__JSONAttribute__c`

Per [`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md) §4.1, this
field has two observed shapes:

**Shape A (modern)** — top-level keyed object:

```jsonc
{
  "ACAT_Phones": [
    {
      "categorycode__c": "ACAT_Phones",
      "categoryname__c": "Mobile Devices",
      "attributeuniquecode__c": "ATT_DT_BRD",
      "attributedisplayname__c": "Brand",
      "valuedatatype__c": "Picklist",
      "value__c": "Infiwave",
      ...
    },
    ...
  ]
}
```

**Shape B (older)** — array of category entries with nested `records`:

```jsonc
[
  {
    "Code__c": "ACAT_X",
    "Name__c": "Category Display Name",
    "productAttributes": { "records": [ { "Code__c", "Name__c", ... } ] }
  }
]
```

The parser:

1. Tries `JSON.parse`. On failure returns `[]` (no attribute data).
2. Inspects the top level. If object, treats as Shape A. If array,
   treats as Shape B.
3. Extracts a normalized `{ code, name, label, dataType, value,
   defaultValue, isRequired }` per attribute, using a fallback chain
   of field names because both shapes use different conventions.
4. The `dataType` extraction in particular searches five candidate
   keys (`valuedatatype__c`, `attributerunningdatatype__c`,
   `dataType`, `inputType`, `attributeRunTimeInfo.dataType`) before
   giving up.

### 5.4 Surfacing the `ATT_DT_` / `ATT_RT_` observation

The parser does *not* automatically classify attributes as
design-time vs run-time based on the prefix convention described in
[`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md) §4.2 — it
returns the codes verbatim. The convention is preserved in the data
that callers can see, but no opinion is encoded into the tool itself.
The reasoning: the convention is observed in standard Vlocity samples
but is not formally documented, so building it into a tool would
imply an authority we do not have.

### Returned shape (sketch)

```jsonc
{
  "detectedNamespace": "hybrid",
  "requestedMode": "auto",
  "queried": ["vlocity_cmt", "standard"],
  "productId": "01t...",
  "details": { ... },
  "vlocity_cmt": {
    "id": "01t...", "name": "...", "specificationType": "Offer",
    "objectClass": {
      "id": "...",
      "name": "Handset",
      "path": [ { "id", "name" }, ... ]
    },
    "attributeCategories": [ { "code", "name", "attributes": [...] } ],
    "attributeMetadataPresent": true,
    "jsonAttributePresent": true
  },
  "standard": {
    "id": "01t...", "name": "...",
    "categories": [ { "id", "name", "catalogId", "catalogName" } ],
    "attributes": [ { "name", "label", "dataType", ... } ]
  }
}
```

### Edge cases handled

- ObjectClass parent loops (capped at 10 hops with no further query)
- Inactive products: returned but the `isActive: false` flag is
  preserved verbatim (callers may choose to filter)
- `ProductAttributeDefinition` missing on a "standard" org: returns
  the product with `attributes: []` rather than failing the whole call
- Both namespaces returning the same `Product2` ID on hybrid orgs:
  expected; each side simply reports what it can see

## 6. Tool selection guidance

The intended order of use:

1. **`namespace_detect`** — always run first against a new org.
   Treat its output as the source of truth for downstream behavior.
2. **`list_products`** — walk the catalogs to find candidate products
   worth inspecting. The default `productLimit: 200` is enough to get
   a feel for an org without overwhelming the LLM context.
3. **`get_product_details`** — deep dive into a specific product
   surfaced by `list_products`.

LLM clients invoking these tools generally do not need to know the
internals of the SOQL — they can chain the tools by extracting IDs
from one result and feeding them into the next.

## 7. Error handling and observability

- Tool errors are returned with `isError: true` and a single text
  content block describing the cause. The JSON payload shape is never
  half-filled on error.
- Salesforce API errors propagate their messages through (e.g.
  permission denied messages are passed back unchanged so the caller
  can see *which* permission is missing).
- SOQL injection guards reject inputs before any query is built; no
  partial-execution state is possible.
- Connection caching means an initial slow call is followed by faster
  subsequent calls within the same process.

## 8. What is *not* in scope (and why)

The following are intentionally omitted from this suite. Each is
genuinely outside what a read-only inspection tool should do.

| Capability | Why not in scope |
|------------|------------------|
| Any write or generation against a Salesforce org | This suite is read-only by design; write tooling raises the safety surface considerably and is outside its purpose |
| Runtime pricing evaluation | Computing a final price requires `PriceListEntry__c` / `PriceAdjustmentSchedule` / `PricingRecipe` traversal and runtime context — out of scope for read-only structural inspection |
| Runtime orchestration walking | Inspecting `OrchestrationPlan__c` *instances* (not their definitions) is runtime-bound and outside the static inspection scope; see [`NAMESPACE_TRANSITION.md`](./NAMESPACE_TRANSITION.md) §2.4 |
| Context rule evaluation | Read-only evaluation against runtime context is complex and runtime-bound; out of scope for static inspection |
| Promotions / discounts walking | Mostly TBD on the Core side; revisited once mapping is clearer |
